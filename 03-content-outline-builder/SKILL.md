# content-outline-builder

## 核心定义

- Outline 是内容大纲，是内容策划层的核心产物。
- Outline 解决"文章按哪些章节展开、每节回答什么问题、每节写哪些关键内容"。
- 本 Skill 负责把已确定选题改写成可确认、可成稿、可拍摄的结构化大纲。
- 本 skill 的主产物是 `content-outline.md`，必须包含可直接确认和进入写作的文章大纲。
- 执行前读取 `../_shared/knowledge-base-index.md`、`../_shared/account-mode-guide.md`、`../_shared/content-module-map.md`、`../_shared/style-guide.md` 和 `../_shared/intermediate-format.md`，保证资料、账号边界和产物格式一致。

## 触发条件

- 用户说"整理大纲"、"生成大纲"、"生成 outline"
- 用户说"先定文章结构"、"这个题怎么写"
- 用户提供了明确选题，想要展开成可写结构
- 用户需要确认核心判断、切入角度和章节安排

**不触发：**
- "帮我找选题" / "抓热点" → 生成选题（`topic-generator`）
- "我有想法但说不清楚" → 生成选题（`topic-generator`）
- "写文章" / "帮我写公众号" → 大纲扩写（`outline-expander`）

## 不做什么

- 不写正文
- 不设计标题方向、开头草稿、结尾草稿
- 不做选题发现
- 不做包装改稿
- 不做发布

## 执行流程

### Step 1: 接收选题

从以下来源接收：
- 用户直接输入的一句话选题
- topic-pool.md 中的某个选题条目
- topic-ranking.md 中推荐的 Top 选题
- topic-material-pack.md 中的访谈素材
- 用户已有的素材、观点、产品说明或草稿片段

优先接收带资料依据的 `topic-ranking.md` 或 `topic-material-pack.md`。如果只有选题标题而没有账号模式、核心判断和可用资料，先退回 `topic-generator` 补资料。

如果方向已明确但交接字段不足，每轮只确认 1 个最高优先级缺口，并给出 2-4 个选项。优先顺序为：核心判断 -> 目标读者 -> 写作目标。不要一次性让 Emma 填多个字段。

### Step 2: 补全素材

读取 `references/research-methods.md`：
1. 先从用户提供的信息和知识库片段中提炼
2. 不够时联网搜索（可选，不阻塞）
3. 仍不够时知识推断兜底

每条素材必须通过 相关性/具体性/可验证/冲突感 四项检查。

### Step 3: 锁定方向

读取 `references/outline-quality-guide.md`。

必须先锁定：
- 目标读者：具体到一类人，不接受"所有人"
- 核心判断：一句话、有立场、可被文章反复支撑
- 写作目标：希望读者读完理解、相信或行动什么
- 切入角度：不能是维基百科式介绍，要说明"这对读者有什么用"

### Step 4: 生成 SCQA

必须输出完整 SCQA：
- 情境(S)：读者熟悉的场景
- 冲突(C)：场景中的矛盾
- 问题(Q)：基于冲突的核心问题
- 答案(A)：文章要给出的解决方案或观点

### Step 5: 生成文章大纲

必须输出独立字段或章节，标题明确叫 `文章大纲`。

大纲要求：
- 5-10 个一级章节
- 每个章节必须包含：
  - 章节标题
  - 本节要回答的问题
  - 本节要写的关键内容
  - 可使用的素材
- 章节顺序必须能看出文章推进逻辑
- 不接受只有"问题 → 方法 → 总结"这种抽象框架
- `推荐框架` 只说明文章类型，不能替代 `文章大纲`

如果是软件介绍、教程、工具使用类选题，大纲必须覆盖实际使用路径，例如：
- 安装/入口
- 初始化配置
- 核心操作
- 典型工作流
- 权限/边界
- 维护方法

如果是观点类文章，大纲必须覆盖：
- 问题提出
- 核心判断
- 论据展开
- 反方/风险
- 行动建议

### Step 6: 输出文件

读取 `../_shared/intermediate-format.md` 中 `content-outline.md` 的格式规范，创建 Markdown 文档，不要在聊天中输出代码块。

默认输出到本地文件：
`content-production/内容大纲/<用户选择的标题>.md`

文件名使用用户最终选择或确认的选题标题；不要使用英文前缀、英文 slug 或 `content-outline-`。如果同一标题需要多个阶段文档，用中文阶段目录区分。

如果用户指定保存路径，以用户路径为准。用户指定完整 `.md` 文件路径时直接保存到该文件；用户指定目录时，保存为 `<用户指定目录>/<用户选择的标题>.md`。

### Step 7: 质量自检

读取 `references/outline-quality-guide.md` 中的自检要求，至少检查：
1. 核心判断够硬？
2. 读者够具体？
3. 素材够硬？（至少 3 条，每条标注可信度）
4. SCQA 完整？
5. 文章大纲明确？
6. 大纲是否覆盖题目承诺？
7. 有风险意识？

任何一项不通过，回去改。

### Step 8: 用户确认

展示 outline 摘要，等待用户确认：
- 核心判断
- 切入角度
- 文章大纲
- 素材使用方式

如果生成了多个大纲方案供选择，聊天里用纵向对比表展示方案差异；用户确认后，只把最终选中的方案写入 Markdown 文档。

有修改意见则更新后重新确认。

## Guardrails

- 不再输出旧版选题简报；历史资料统一整理成 content-outline。
- 必须包含明确的文章大纲；不能用"推荐框架"冒充大纲
- 必须包含 SCQA 四个部分
- 素材至少 3 条，并标注可信度；涉及医学判断的核心依据不足时，不能用推断补成事实
- 必须包含风险提醒
- 软件介绍类文章必须写出具体使用路径
- 生成大纲后必须停下来，让用户确认结构、角度和素材使用方式
- 只有用户明确确认“大纲没问题”“继续写正文”等同义表达后，才进入大纲扩写（`outline-expander`）
- 联网搜索失败时用知识推断兜底，不阻塞输出
